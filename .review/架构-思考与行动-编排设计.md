# 架构重梳理：共享 State · 并发 Think · 顺序 Act（LangGraph 编排）

> 按你的模型重新捋清：**“思考”并发、依状态而变；“State”是共享信息；“行动”由引擎排布、顺序执行。**
> 多人行动分两类：**抢占式（并发→仲裁）** 与 **排队式（顺序→边走边看）**。本文先讲逻辑，再据此实现。

## 1. 三个基本概念

| 概念 | 性质 | 谁动它 | LangGraph 对应 |
|---|---|---|---|
| **State 共享状态** | 单一事实来源（牌局全量：席位/阶段/回合/事件日志/pending） | 只有引擎在“行动”时改 | 图的 **State 通道** `gs` |
| **Think 思考** | **并发**、依当前 State 而变；产出“意图”+可选“发言” | 不改 State | `Send` **并发扇出**到 `think_one` |
| **Act 行动** | 引擎**排布、顺序、确定性、可校验** | 引擎（唯一写 State 处） | `arbitrate` / `queued_phase` |

关键不变量：**Think 只读 State 产出意图；只有 Act 改 State。** 这样多智能体并发思考不会互相污染，
而状态变更始终经引擎仲裁、可复盘。

## 2. 两种多人行动模式（按阶段定）

### 抢占式 Preemptive（并发 → 仲裁）
所有相关 agent 在**同一份冻结快照**上**同时**思考、各自提交意图；彼此**看不到对方本回合的意图**；
引擎按**优先级/计票**统一仲裁。用于秘密同时行动。
- 例：夜晚（狼群投票取多数刀人、预言家查验、女巫救/毒，按 `resolution_order` 仲裁）、白天投票（同时投、统一计票）。
- LangGraph：`dispatch ──Send×N──> think_one(并发) ──> arbitrate(按优先级顺序应用) ──> dispatch`

### 排队式 Queued（顺序 → 边走边看）
agent 按固定顺序行动；每个人在**前人已更新的 State** 上思考，因此后者能看到前者的发言/动作。用于轮流发言。
- 例：白天讨论（按座位轮流发言，后发言者看到前面所有发言）。
- LangGraph：`dispatch ──> queued_phase(按序：思考→发言→写State→下一个) ──> dispatch`

### 阶段 → 模式映射（狼人杀）
| 阶段 | 模式 | 原因 |
|---|---|---|
| night | 抢占式 | 狼/预言家/女巫同时秘密行动，引擎按 resolution_order 仲裁 |
| day_discussion | 排队式 | 轮流发言，后者看到前者的话 |
| day_vote | 抢占式 | 同时投票，引擎统一计票 |

判定规则：阶段动作含 `speak` → 排队式；否则 → 抢占式。

## 3. 编排图（外层）

```
START → dispatch ─┬─ finished ─────────────────────────────→ END
                  ├─ 无人可动 → advance(引擎结算+转阶段) ────→ dispatch
                  ├─ 抢占式  → Send×N → think_one(并发思考) → arbitrate(顺序仲裁) → dispatch
                  └─ 排队式  → queued_phase(轮流发言/行动) ───────────────────→ dispatch
```

节点职责：
- **dispatch**（条件路由）：读 State，判断当前阶段 mode 与待行动席位，决定走哪条边。
- **think_one(seat)**：在冻结快照上为该席位产出**意图**（`Action`），**不改 State**；并发分支共享同一快照 → 无信息泄露。
- **arbitrate**：把收集到的意图按**优先级/座位序**交给引擎 `apply`（夜晚的真正结算在 `advance` 按 `resolution_order` 做；投票在 `advance` 计票）。**唯一写 State**。
- **queued_phase**：顺序遍历未行动席位；每个 agent **思考 → 生成发言(LLM 文本) → `apply`(speak 事件写入共享 log) → 下一个**；中途遇人类则停下等输入。
- **advance**：引擎 `advance_phase`（结算、判胜负、转阶段）。

State 通道：`gs`（共享，引擎改）、`intentions`（`Send` 并发收集，仲裁后重置）、`steps`（累加，防超步）。

## 4. 按角色分支（role branching）

`think_one` 内部按角色走不同决策路径——这由引擎的 `legal_actions(state, seat)` 天然给出：
狼→`eliminate`、预言家→`investigate`、女巫→`protect`/`eliminate`、平民→`vote`/`speak`。
LLM 决策链据**角色 persona + 共享 State + 自身心证**在合法集合里选择，并（讨论阶段）生成发言文本。

## 5. 发言（“多个 AI agent 轮流发言”——可玩性关键）

排队式讨论里，每个 agent 用 LLM 依据**共享 State + 自己的身份/心证**生成**一句发言**，作为 `speak` 事件
（带 `text`，public 可见）写入共享 log；下一个发言者读到 → 形成真实的多智能体讨论，而非空过场。

## 6. 人类在环（human-in-the-loop）

人类是 actors 之一：
- 排队式：轮到人类席位 → 暂停，等前端提交（发言/过）；其余 AI 在其前后按序进行。
- 抢占式：AI 在冻结快照上各自提交（看不到人类意图），人类亦提交其秘密意图，引擎统一仲裁。
- HTTP 实现：`PlayService` 在“轮到人类”时停下返回视图；人类提交后继续驱动 AI 到下一次人类回合或终局。
（编排器与 PlayService 复用同一套 `phase_mode / collect_preemptive / run_queued` 步进函数，保证全 AI 与人机两条路径一致。）

## 7. 为什么这样切分

- **思考并发**：贴合“同时秘密行动”的真实语义，且可并行省时（LLM 调用可并发）。
- **行动顺序**：状态变更集中在引擎、确定性、可校验、可复盘，杜绝竞态。
- **两种模式**：把“同时暗投”与“轮流明说”这两类多人交互显式建模，正是狼人杀（及多数社交推理游戏）的骨架。
- **共享 State**：所有 agent 看同一份信息（按可见性裁剪），是“多智能体协作/对抗”的基础。

## 9. 本轮"按思路重构"的改动（整理统一）

把整个项目收敛到本设计，去掉冗余实现，并修掉导致"不可玩"的根因：

1. **单一编排出口**：`game_graph`（旧的线性 play_phase 自循环图）改为 `orchestrator` 的**兼容别名**，
   全项目（autoplay / PlayService）统一走 `orchestrator`（抢占式并发思考 + 排队式发言）。`app.agents.__init__`
   按"内层单 agent 思考 / 外层多 agent 编排"重新组织导出。
2. **修复"人类#0 几乎每晚必死"**（不可玩根因）：引擎夜晚群体击杀**平票不再固定取最小席位**，改用本局
   种子做确定性随机（`GameState.seed`）。此前平票总落到 0 号 → 人类（座位 0）几乎每局夜 1 被刀、无法参与。
3. **人类也参与排队发言**：前端牌桌为人类的 `speak` 行动提供输入框（回车/按钮发言），人类作为 agent 之一
   与 AI **轮流发言**；后端 `action.extra.text` 贯通到事件日志。
4. **验证**：后端 `uv run pytest` → **106 passed**（含人类发言持久化、夜杀平票不固定 0 号等用例）；
   前端 `npm run build` 通过；实测人类存活到讨论即可发言、AI 轮流发言、按角色秘密行动。
